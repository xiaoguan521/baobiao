# 配置使用手册

## 1. 项目用途

本项目用于按指定月份从 Oracle 表 `PT_DXSL_2106032001_05` 查询基础数据，并写入 Excel 模板中的基础值单元格。

设计原则：

- 只写基础数据
- 不覆盖模板中的公式单元格
- 支持整本输出
- 支持单个 sheet 输出
- 支持 CLI、HTTP、Docker 三种使用方式
- 业务口径和网点映射尽量放到配置文件中维护

## 5. HTTP 接口说明

### 5.1 健康检查

```http
GET /api/health
```

返回示例：

```json
{"ok":true}
```

### 5.2 查看支持的 sheet

```http
GET /api/reports/sheets
```

返回示例：

```json
{
  "sheets": ["总表", "市本级", "东港", "凤城", "宽甸", "排名", "占比", "各渠道复核业务来源统计", " 复核业务量统计"],
  "configPath": "/Users/xiaochen/Desktop/123/config/report-rules.json"
}
```

### 5.3 生成报表

```http
POST /api/reports/generate
Content-Type: application/json
```

请求体字段：

| 字段 | 必填 | 说明 |
|---|---|---|
| `month` | 是 | 月份，格式 `YYYY-MM` |
| `sheetName` | 否 | 指定 sheet 名称 |
| `sheetOnly` | 否 | 是否只输出该 sheet |
| `outputPath` | 否 | 指定输出文件 |
| `outputDir` | 否 | 指定输出目录 |
| `templatePath` | 否 | 指定模板路径 |
| `user` | 否 | Oracle 用户 |
| `password` | 否 | Oracle 密码 |
| `dsn` | 否 | Oracle 连接串 |

整本示例：

```json
{
  "month": "2025-12"
}
```

单 sheet 示例：

```json
{
  "month": "2025-12",
  "sheetName": "排名",
  "sheetOnly": true,
  "outputPath": "/Users/xiaochen/Desktop/123/generated/api_ranking_2025-12.xlsx"
}
```

### 5.4 下载文件

```http
GET /api/reports/download?path=/Users/xiaochen/Desktop/123/generated/api_ranking_2025-12.xlsx
```

### 5.5 查看未匹配网点调试信息

```http
GET /api/reports/debug/unmatched?month=2025-12&limit=20
```

这个接口用于找出：

- 哪些经办人没有匹配到网点
- 哪些 `dxms` 描述没有匹配到网点
- 哪些地区未匹配量最高
- 哪些业务事项未匹配量最高

适合用来持续补 `outletRules`。

### 5.6 接口鉴权说明

当设置环境变量 `REPORT_API_TOKEN` 后，以下接口需要带 token：

- `POST /api/reports/generate`
- `GET /api/reports/debug/unmatched`
- `GET /api/reports/download`

支持两种传法：

```http
Authorization: Bearer your-token
```

或：

```http
X-API-Token: your-token
```

说明：

- `GET /api/health` 和 `GET /api/reports/sheets` 默认不鉴权
- 下载接口只能访问 `DOWNLOAD_ROOT` 目录内的文件

## 6. 核心配置文件说明

配置文件位置：

```text
config/report-rules.json
```

### 6.1 `businessRules`

作用：把 Oracle 中的业务名称归类到模板口径。

字段说明：

| 字段 | 说明 |
|---|---|
| `sheetGroup` | 所属大类，如 `归集业务` |
| `summaryGroup` | 汇总大类，如 `归集`、`提取`、`贷款`、`贷后` |
| `item` | 模板里的具体事项名称 |
| `patterns` | 关键字数组，命中任意一个就算匹配 |

示例：

```json
{
  "sheetGroup": "提取业务",
  "summaryGroup": "提取",
  "item": "房租提取",
  "patterns": ["租房提取申请", "房租提取"]
}
```

### 6.2 `regionNameByCode`

作用：把库里的地区编码映射成模板里的地区名。

示例：

```json
{
  "1": "市本级",
  "2": "东港",
  "3": "凤城",
  "4": "宽甸"
}
```

### 6.3 `centerNameByRegion`

作用：配置每个地区“中心”对应的模板网点名。

示例：

```json
{
  "市本级": "公积金大厅",
  "东港": "东港分部"
}
```

### 6.4 `outletRules`

作用：把明细记录匹配到具体网点。

这是最关键、也最需要持续维护的一块。

字段说明：

| 字段 | 说明 |
|---|---|
| `name` | 模板中的网点名称 |
| `patterns` | 对 `dxms` 做正则匹配 |
| `creators` | 对经办人 `dx_05_dxcjzmc` 做直接匹配 |

示例：

```json
{
  "name": "丹行新桥支行",
  "patterns": ["丹东银行.*新桥支行", "丹行.*新桥支行", "新桥支行"],
  "creators": []
}
```

如果后面确认某个经办人长期属于某个网点，可以这样加：

```json
{
  "name": "工行锦山支行",
  "patterns": ["工商银行.*锦山支行", "工行.*锦山支行"],
  "creators": ["张三", "李四"]
}
```

如果这批经办人更像中心柜面或网厅经办人，也可以直接归到地区中心：

```json
{
  "name": "公积金大厅",
  "patterns": [],
  "creators": ["互联网用户", "陈俊霖", "刘雯"]
}
```

建议把这类“中心规则”放在各地区银行网点规则后面，这样带明确支行关键字的记录会优先命中银行网点，再由中心规则兜底。

### 6.5 `bankNameByOutlet`

作用：把网点归到银行，用于“总表”“占比”等银行级统计。

### 6.6 `channelBuckets`

作用：把库里的渠道值归并到模板里的渠道列。

例如：

- `柜台业务`、`柜面`、`柜台` -> `柜面`
- `网厅`、`网厅业务`、`网上柜台` -> `网厅`
- `app`、`政务app`、`yjs_zw` -> `政务一件事`

### 6.7 `reviewerNames`

作用：指定“复核业务量统计”sheet 里需要统计的复核人名单。

### 6.8 `reportSheets`

作用：控制外部接口允许导出的 sheet 名单。

## 7. 常见配置调整场景

### 场景 1：某个业务事项没有统计进去

处理方式：

1. 查看原始 `dxmc` 或 `dxms`
2. 在 `businessRules` 中给对应 `item` 增加关键字
3. 重新生成报表

### 场景 2：某个网点一直没有匹配上

处理方式：

1. 执行：

```bash
node src/cli.js --month 2025-12 --debug-unmatched --limit 20
```

2. 看 `unmatchedCreators` 和 `unmatchedDxms`
3. 在 `outletRules` 中补 `patterns` 或 `creators`
4. 重新生成报表

### 场景 3：换了新模板路径

处理方式：

- CLI 用 `--template`
- HTTP 用 `templatePath`
- 或修改环境变量 `TEMPLATE_PATH`

## 9. 当前已知限制

1. 现在的网点匹配规则还不完整，特别是“市本级”部分很多记录仍未能准确落到具体网点。
2. 当前程序优先保证“公式不被覆盖”和“结构可持续维护”，不是最终业务口径的绝对定稿。
3. 如果后续模板结构改动较大，部分单元格定位可能需要同步调整。

## 10. 推荐维护流程

建议每次新月份上线时按这个流程走：

1. 先运行单月生成
2. 再运行未匹配调试
3. 补 `config/report-rules.json`
4. 再次生成并人工抽查关键 sheet
5. 确认后用于正式输出

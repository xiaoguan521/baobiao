# report-rules 配置说明

## 1. 文件位置

默认配置文件：

```text
config/report-rules.json
```

也可以通过环境变量覆盖：

```text
REPORT_RULES_PATH
```

这个文件决定了：

- 业务事项怎么分类
- 地区代码怎么映射
- 明细怎么归属到网点
- 网点怎么归属到银行
- 渠道怎么归并
- 复核人名单怎么统计
- 哪些 sheet 允许导出

## 2. 最常改的配置

### 2.1 `businessRules`

作用：

- 把 Oracle 里的 `dxmc`、`dxms` 归类到报表口径

字段：

| 字段 | 说明 |
|---|---|
| `sheetGroup` | 所属业务大类 |
| `summaryGroup` | 汇总分组，常见值为 `归集`、`提取`、`贷款`、`贷后` |
| `item` | 模板里的事项名称 |
| `patterns` | 关键字数组，只要 `dxmc + dxms` 包含任意一个就命中 |

示例：

```json
{
  "sheetGroup": "提取业务",
  "summaryGroup": "提取",
  "item": "房租提取",
  "patterns": ["租房提取申请", "房租提取"]
}
```

适合什么时候改：

- 某个业务事项没有统计进去
- 某类记录被归到错误事项

修改建议：

- 先用最稳定、最能区分业务的关键字
- `patterns` 尽量避免过短，防止误匹配
- 同一事项可以放多个近义关键字

### 2.2 `outletRules`

作用：

- 把一条明细匹配到具体网点

结构：

- 第一层按地区分组，例如 `市本级`、`东港`
- 每个地区下面是一组网点规则

字段：

| 字段 | 说明 |
|---|---|
| `name` | 模板中的网点名称 |
| `patterns` | 对 `dxms` 做正则匹配 |
| `creators` | 对经办人 `dx_05_dxcjzmc` 做直接匹配 |

示例：

```json
{
  "name": "丹行新桥支行",
  "patterns": ["丹东银行.*新桥支行", "丹行.*新桥支行", "新桥支行"],
  "creators": []
}
```

适合什么时候改：

- 某个网点一直匹配不上
- 某些经办人长期固定属于某个网点或中心

修改建议：

- 明确支行关键字规则放前面
- 中心/大厅兜底规则放后面
- `patterns` 用于识别单位或支行名称
- `creators` 用于补充“人名稳定归属”的场景

### 2.3 `bankNameByOutlet`

作用：

- 把网点归到银行
- 影响“总表”“占比”等银行级统计

示例：

```json
{
  "工行锦山支行": "工商银行",
  "农行振兴支行": "农业银行"
}
```

适合什么时候改：

- 新增了网点规则后，需要把它补到银行映射里

### 2.4 `channelBuckets` 和 `channelBucketOrder`

作用：

- 把库里的渠道值归并到模板渠道列

示例：

```json
{
  "柜面": ["柜台业务", "柜面", "柜台"],
  "网厅": ["网厅", "网厅业务", "网上柜台"]
}
```

适合什么时候改：

- 出现新的渠道名称
- 某些渠道被归到了 `其他`

### 2.5 `reviewerNames`

作用：

- 指定“复核业务量统计”里要统计的复核人名单

适合什么时候改：

- 复核人新增、离岗或更名

### 2.6 `reportSheets`

作用：

- 控制哪些 sheet 可以通过接口或页面导出

适合什么时候改：

- 模板新增 sheet
- 某些 sheet 暂时不想对外开放

## 3. 其他基础配置

### 3.1 `regionNameByCode`

作用：

- 把数据库里的地区编码映射为报表地区名

### 3.2 `centerNameByRegion`

作用：

- 为每个地区指定“中心”名称
- 当明细没有匹配到具体银行网点时，会归到这里

### 3.3 `bankOrder`

作用：

- 控制银行展示顺序

### 3.4 `summaryGroupOrder`

作用：

- 控制“归集、提取、贷款、贷后”的展示顺序

### 3.5 `workTime`

作用：

- 定义工作时间、中午时间、非工作时间的分段
- 影响“各渠道复核业务来源统计”

## 4. 最常见的调整流程

### 场景 1：某个事项没有统计进去

1. 先找到原始 `dxmc` / `dxms`
2. 在 `businessRules` 中找到对应事项
3. 补一个更准确的 `patterns`
4. 重新生成报表验证

### 场景 2：某个网点一直没命中

1. 先调用未匹配调试接口：

```text
GET /api/reports/debug/unmatched?month=2025-12&limit=20
```

2. 看 `unmatchedCreators` 和 `unmatchedDxms`
3. 在对应地区的 `outletRules` 下补 `patterns` 或 `creators`
4. 如果是新网点，别忘了同步补 `bankNameByOutlet`

### 场景 3：新增模板 sheet

1. 先确认模板里 sheet 名称
2. 如果后端允许对外导出，就补到 `reportSheets`
3. 如果这个 sheet 需要新口径，还要同步补业务规则或映射规则

## 5. 修改时的注意事项

- 这是 JSON 文件，注意逗号和引号格式
- `patterns` 中很多值会被当作正则处理，像 `.`、`(`、`)` 这类字符要有意识
- 先加更具体的规则，再加更宽泛的兜底规则
- 新增网点后，通常至少要联动检查：
  - `outletRules`
  - `bankNameByOutlet`
  - 必要时 `centerNameByRegion`

## 6. 改完后怎么验证

建议至少做这几步：

1. 重新生成一个月度报表
2. 对照模板检查目标事项或网点是否已命中
3. 再跑一次未匹配调试接口，看未匹配是否下降
4. 检查银行汇总是否受影响
